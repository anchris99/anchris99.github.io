举例理解jvm指令和Java运行时数据区

发表于 2020-10-07 分类于 jvm

举例1

理解：

  1. 首先要知道 jvm stack 中的 栈帧中的 local variables表（局部变量表）和 Operand Stack（操作数栈） 的概念：

    • 每个线程对应一个 jvm stack。
    • 线程中的每个方法对应一个栈帧
    • 栈帧包括四部分： local variables（局部变量表） + operand stack)(操作数栈)+ dynamic linking(动态连接)+ return address（返回地址）。理解这种现象 主要先理解栈帧中的前两个部分。

  2. 对应的指令码的意思：

    • bipush指令对应 将数据压入operand stack，istore是从operand stack中取数据赋值到local variables， 而iload操作是 从 local variables取出数据 加载到操作数栈operand stack。
    • iinc 1 by 1是指 对 i加1操作： Increment local variable by constant 其在官方文档中描述如下：

    Operation

    Increment local variable by constant

    Forms

    iinc = 132 (0x84)

    Operand Stack

    No change

    Description

    The index is an unsigned byte that must be an index into the local variable array of the current frame The const is an immediate signed byte. The local variable at index must contain an int. The value const is first sign-extended to an int, and then the local variable at index is incremented by that amount.

    注意到 该操作是在 local variables表中进行的，并不涉及 操作数栈。

  3. 观察两个案例对应的 指令， 第一个是先 load再 进行iinc，然后又istore， 由于iinc是在local variables中进行的，所以operand stack中的 i值没有改变， 然后又赋值回了 local variables中， 变回了8

  4. 而第二个例子中， 第一个是先iinc再 进行load，然后 store，由于是先进行了iinc操作，所以加载到栈中的 数据已经是加一后 的值（9）， 再赋值回来，仍然是9。

  5. 还有一点： 在Java中看起来从一个局部变量到另外一个局部变量是一步操作完成的，其实在jvm指令码层面是两步， 比如：

对应两个步骤：

  • 首先 从local variables中读取i的值放入 operand stack
  • 从 operand stack中 取值存入 local variables中的 变量

举例2

new 之后发生了什么

对应的字节码指令如下：

new :创建一个新的对象 Create new object， 此时在 operand stack中存放的是 新对象的引用。 新对象此时是默认值（零值）

dup： Duplicate the top value on the operand stack and push the duplicated value onto the operand stack.

invokespecial：调用 特殊方法（这里是初始化方法）（会消耗掉栈顶 duplicate后的元素）

astore_1: 将 operand stack 栈顶元素存入 local variables局部变量表。

由上面的内容也进一步理解了 DCL模式时为什么要加 volatile， 若不加volatile则 可能会发生指令重排序， 那么 astore指令就可能在 invokespecial之前那执行，那么另外一个线程就有可能使用到对象 半初始化的数据（也就是说执行完了 new（此时值是默认值） 尚未执行初始化操作）。因此要用volatile修饰变量。

java内存模型

发表于 2020-10-06 更新于 2020-10-07 分类于 jvm

硬件层面

存储器层次结构

如图所示，现代的计算机存储器是分层次的， 金字塔底下存取速度慢，容量大，成本低， 金字塔上面 存取速度快，容量小，成本高。

保证缓存一致性

在高速缓存中，保存着从主存读取过来的数据，其中 L3缓存是 所有CPU共享， L1，L2缓存属于CPU本身，在CPU内部。

如下图所示， 由于L2和L1为CPU独有，那么如何保证每个CPU中L2和L1中缓存内容的一致性问题呢？

比较古老的CPU里面采用的锁总线的方式，当有一个CPU访问主存数据内容的时候，将　CPU到主存的总线锁住， 不允许其他CPU进行操作。 弊端就是 效率低下，并发度低。

现代CPU以Intel为代表，使用的是MESI（缓存一致性协议）来保证各个缓存之间数据的一致性 。

MESI有四种标记状态：

  • modified :如果自身对数据进行了修改，那么就标记为 modified
  • exlusive: 表明数据独享
  • shared： 数据共享
  • invalid： 如果其他线程对数据进行了操作，置为该标记表明数据失效。再需要使用时，需要去主存重新读取。

缓存行和存在的问题及解决

  • cache line: 读取缓存的时候，是整行读取的，成为cache line,（缓存行）

  • 伪共享： 位于同一缓存行的数据，被两个不同的CPU使用，造成数据不必要的失效，重新读取，使得性能下降的现象。（一个CPU使用的只是 缓存行中的一个数据x，另一个CPU使用缓存行中的数据y， x和y在同一行中， 一个CPU仅仅操作了x，结果导致整个缓冲行失效，所以y必须重新读取（然而y的值并没有被改变，这就导致了不必要的操作）

  • 解决方法：

    • java8中增了一个注解：@sun.misc.Contended。加上这个注解的类会自动补齐缓存行。

    • 使用数据进行填充。占满整个缓冲行 （64字节）

  • 缓存对齐可以提高效率，当然也牺牲掉了空间

指令重排序

为了提高指令的执行效率， Java 会将指令进行重新排序， 但有时候会造成错误。

解决指令重排序可以使用 volatile进行禁止指令重排序。

底层原理

硬件内存屏障 X86

sfence: store| 在sfence指令前的写操作当必须在sfence指令后的写操作前完成。
lfence：load | 在lfence指令前的读操作当必须在lfence指令后的读操作前完成。
mfence：modify/mix | 在mfence指令前的读写操作当必须在mfence指令后的读写操作前完成。

原子指令，如x86上的”lock …” 指令是一个Full Barrier，执行时会锁住内存子系统来确保执行顺序，甚至跨多个CPU。Software Locks通常使用了内存屏障或原子指令来实现变量可见性和保持程序顺序

Java内存模型

所有的变量都存储在主内存中，每个线程还有自己的工作内存，工作内存存储在高速缓存或者寄存器中，保存了该线

程使用的变量的主内存副本拷贝。

线程只能直接操作工作内存中的变量，不同线程之间的变量值传递需要通过主内存来完成

保障可见性

由于每个线程都有自己的工作内存，这些不是共享的，为了保证数据更改时能及时更新数据，保证数据在线程之间的 内存可见性， 需要加入volatile关键字，（底层用 lock实现）

lock细节

  1. 使用了volatile 在 底层 对应的是lock指令 可以加入 CPU级别的内存屏障，屏障两侧的指令不可以重排序。
  2. lock指令的作用：
    • 立即将数据同步回主内存
    • 触发MESI缓存一致性协议，相应线程对应的数据失效。
    • 相关线程在 unlock后再去读取最新的数据。（保证读取到的是更新后的值而非原来的旧值）
