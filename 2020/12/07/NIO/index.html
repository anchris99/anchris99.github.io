网络IO模型

网络iO模型主要分为：

  • 同步阻塞io
  • 同步非阻塞io
  • 多路复用io
  • 异步io
  • 信号驱动型io

网络传输数据的过程主要分为两步： 等待数据+操作

根据操作数据的过程是否阻塞，分为异步和同步io

  • 操作时 阻塞： 同步
  • 操作时 不阻塞：异步

根据 等待数据时候的 线程状态：分为：

  • 同步阻塞： 等待的时候一直阻塞

  • 同步非阻塞：等待数据的时候 采用轮询的方式

  • 多路复用： 在select/poll/epoll的 帮助下，同时监控多个过程，当有过程的数据等待完成的时候，就通知进行操作。

Java nio

BIO

传统的Java io是采取的同步阻塞的方式，也就是上面的 第一种网络模型。

优点：

活动连接数不是特别高（小于单机1000）的情况下，这种模型是比较不错的，可以让每一个连接专注于自己的I/O并且编程模型简单，也不用过多考虑系统的过载、限流等问题。

缺点：

  1. 每连接每线程：为每个连接都开一个线程来处理。

  2. 线程的创建和销毁成本很高，在Linux这样的操作系统中，线程本质上就是一个进程。创建和销毁都是重量级的系统函数。

  3. 线程本身占用较大内存，像Java的线程栈，一般至少分配512K～1M的空间，如果系统中的线程数过千，恐怕整个JVM的内存都会被吃掉一半。

  4. 线程的切换成本是很高的。

NIO

概述

NIO 即 non-blocking io或者 new io， 是对 bio不足之处的改进版本。

NIO基于 网络IO模型中的 多路复用模型， 主要由三部分组成：

  • selector
  • channel
  • buffer

区别

  1. NIO是以块的方式处理数据，但是IO是以 字节流的形式去写入和读出的。
  2. NIO不再是和IO一样用OutputStream和InputStream 输入流的形式来进行处理数据的，但是又是基于这种流的形式，而是采用了通道和缓冲区的形式来进行处理数据的。
  3. 还有一点就是NIO的通道是可以双向的，但是IO中的流只能是单向的。
  4. 还有就是NIO的缓冲区可以进行分片，可以建立只读缓冲区、直接缓冲区和间接缓冲区。
  5. NIO同步非阻塞，IO同步阻塞。

基本工作过程

NIO的线程模型对Socket发起的连接不需要每个都创建一个线程，完全可以使用一个Selector来多路复用监听N多个Channel是否有请求。

这里面是基于操作系统底层的Select通知机制的，一个Selector不断的轮询多个Channel，这样避免了创建多个线程。

可能说1000个请求， 只有100个请求是有数据交互的，可能server端就提供10个线程就能够处理这些请求。这样的话就可以避免了创建大量的线程。

对比传统socket

每次一个客户端接入，都是要在服务端创建一个线程来服务这个客户端的

这会导致大量的客户端的时候，服务端的线程数量可能达到几千甚至几万，几十万，这会导致服务器端程序负载过高，不堪重负，最终系统崩溃死掉。

Buffer

概述

  • Buffer 是一个对象， 它包含一些要写入或者刚读出的数据。在 NIO 中加入 Buffer 对象，体现了新库与原 I/O 的一个重要区别。在面向流的 I/O 中，您将数据直接写入或者将数据直接读到 Stream 对象中
  • 在 NIO 库中，所有数据都是用缓冲区处理的。在读取数据时，它是直接读到缓冲区中的。
  • 缓冲区实质上是一个数组。通常它是一个字节数组。

selector

Why Use a Selector?

简而言之：在selector的帮助下，可以使单个线程处理多个 channel.减小资源的浪费。

The advantage of using just a single thread to handle multiple channels is that you need less threads to handle the channels. Actually, you can use just one thread to handle all of your channels. Switching between threads is expensive for an operating system, and each thread takes up some resources (memory) in the operating system too. Therefore, the less threads you use, the better.

Keep in mind though, that modern operating systems and CPU’s become better and better at multitasking, so the overheads of multithreading becomes smaller over time. In fact, if a CPU has multiple cores, you might be wasting CPU power by not multitasking. Anyways, that design discussion belongs in a different text. It suffices to say here, that you can handle multiple channels with a single thread, using a Selector.
