MVCC

发表于 2020-09-29 分类于 数据库

概念

MVCC (Multiversion Concurrency Control) 中文全程叫多版本并发控制， 是数据库存储引擎实现中常用的处理读写冲突的手段，目的在于提高数据库高并发场景下的吞吐性能。

MySQL 中的 MVCC 实现，从 《高性能 MySQL》一书中对 MVCC 的介绍可知：

  1. MySQL 中 InnoDB 引擎支持 MVCC
  2. 应对高并发事务, MVCC 比单纯的加行锁更有效, 开销更小
  3. MVCC 在读已提交（Read Committed）和可重复读（Repeatable Read）隔离级别下起作用
  4. MVCC 既可以基于乐观锁又可以基于悲观锁来实现

实现

MVCC的实现，是通过保存数据在某一个结点的快照实现的。

MVCC只应用 在 读已提交和可重复读 隔离级别， 而 未提交读 隔离级别总是读取最新的数据行，无需使用MVCC。可串行化隔离级别需要对 所有读取的行都加锁，单纯使用 MVCC 无法实现。

不同的存储引擎，其MVCC的实现原理也不同， 这里主要学习 主流存储引擎InnoDB的实现方式。

MVCC 在每行记录后面都保存着两个隐藏的列，用来存储两个版本号：

创建版本号：指示创建一个数据行的快照时的系统版本号；

删除版本号：如果该快照的删除版本号大于当前事务版本号表示该快照有效，否则表示该快照已经被删除了

版本号： 标志当前版本的连续递增的号码， 每开启一个事务，系统版本号都会递增。

SELECT

  1. InnoDB只查找 版本早于 当前事务版本的数据行 ，这样可以确保事务读取的行，要么是在事务开始前已经存在的，要么是事务自身插入或者修改过的。
  2. 行的删除版本要么未定义，要么大于当前事务版本号。（小于等于的话说明已经被删除了） 这可以确保事务读取到的行，在事务开始之前未被删除。

只有符合上述两个条件的记录，才能返回作为查询结果

INSERT

InnoDB为新插入的每一行保存当前系统版本号作为行版本号。

DELETE

InnoDB为删除的每一行保存当前系统版本号作为行删除标识。

UPDATE

InnoDB为插入一行新记录，保存当前系统版本号作为行版本号，同时保存当前系统版本号到原来的行作为行删除标识。

可以理解为 新 插入一条数据，同时作废（删除） 之前旧的数据。

MVCC优缺点

优点

由于MVCC的 存在，使得很多操作不用加锁就可以正确的完成，简单高效， 极大的提高了并发的效率。

缺点

由于要为每行记录都增加 创建和删除版本号， 所以增加了存储空间

要进行版本号的比较，需要做额外的 维护工作和 查询比较工作。

乐观锁悲观锁

上面提到，不同的存储引擎其MVCC实现思想是不一样的，有悲观锁，有乐观锁，那么二者有什么区别呢？

  • 悲观锁（Pessimistic Lock），顾名思义，就是很悲观，每次去拿数据的时候都认为别人会修改，所以每次在拿数据的时候都会上锁，这样别人想拿这个数据就会block直到它拿到锁。假定会发生并发冲突，事先屏蔽一切可能违反数据完整性的操作。
  • 乐观锁（Optimistic Lock），顾名思义，就是很乐观，每次去拿数据的时候都认为别人不会修改，所以不会上锁，但是在提交更新的时候会判断一下在此期间别人有没有去更新这个数据。乐观锁适用于读多写少的应用场景，这样可以提高吞吐量。乐观锁假设不会发生并发冲突，只在提交操作时检查是否违反数据是否和预期的一致

当前读快照读

  1. 快照读： 读取的是 快照中的内容， 不需要加锁。

使用场景：

单纯的select操作，不包括

Read Committed隔离级别：每次select都生成一个快照读。

Read Repeatable隔离级别：开启事务后第一个select语句才是快照读的地方，而不是一开启事务就快照读。

  1. 当前读： 读取的是当前内容，需要加锁

使用场景：

因为必须获取 最新的信息，所以必须 使用 当前读。

实现方式： next -key -locks, 由两部分组成 gap_lock 和 record_lock组成。

record lock 锁定记录上的索引
gap lock 锁定 索引间的间隙，不包括索引本身
next -key -locks 由record lock+gap lock组成， 既锁定索引本身也锁定索引的间隙

解决幻读

MVCC并没有从根本上解决 幻读的问题， MVCC+next_key_locks 二者一起可解决 幻读的问题。

幻读： T1 读取某个范围的数据，T2 在这个范围内插入新的数据，T1 再次读取这个范围的数据，此时读取的结果和和第一次

读取的结果不同。

由于有next_key_locks的存在，T1读取某个范围的时候，T2不能对其进行修改，所以解决了幻读这个问题。

实际上就是 MVCC+当前读， 这样就不会产生读取的时候被插入新数据的情况了。
