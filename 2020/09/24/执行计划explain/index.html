执行计划explain

发表于 2020-09-24 更新于 2020-09-27 分类于 数据库

什么是explain

EXPLAIN 关键字可以模拟优化器执行 SQL 查询语句，根据返回信息得知 MySQL 是如何处理对应的SQL语句的。 进而可以对语句的性能进行分析优化。

用法： Explain+SQL 语句。

这些表里的选项都是些什么意思呢？

表格分析

id

表示查询中执行 select 子句或操作表的顺序。

  • id相同时， 执行顺序从上到下
  • id不同时， id越大，越先执行。
  • id相同和不同的，同时存在：相同的可以认为是一组，从上往下顺序执行，在所有组中，id值越大，优先级越高，越先执行。

select_type

select_type 代表查询的类型，主要是用于区别普通查询、联合查询、子查询等的复杂查询。

Value Meaning
SIMPLE 简单的select 不包括UNION或者子查询
PRIMARY 子查询的外层被标
UNION 第二个 select 出现在union之后
SUBQUERY 在SELECT或WHERE列表中包含了子查询
UNCACHEABLE SUBQUERY A subquery for which the result cannot be cached and must be re-evaluated for each row of the outer query
DERIVED 在 FROM 列表中包含的子查询被标记为 DERIVED
UNION RESULT 从UNION表获取结果的SELECT

table

对应行正在访问哪一个表，表名或者别名，可能是临时表或者union合并结果集

type

访问类型， 表示我是以何种方式去访问我们的数据 ，性能从高到低对应依次是：

常用的几个：

一般情况下，得保证查询至少达到range级别，最好能达到ref.

  • system:系统级别
  • const:常量级别，表示通过索引一次就找到了。
  • ref: 返回所有匹配某个单独值的行， 使用了非唯一性索引扫描 ，本质上也是索引访问。
  • range:范围访问，只检索给定范围的行， 一般是使用了between > <等范围约束条件。
  • inedex:使用了覆盖索引或者是利用索引进行了排序分组。
  • ALL:全表扫描

key相关

  • possible keys: 可能用到的索引。
  • key： 实际用到的 索引
  • key_len : 索引中使用的字节数， 越长，说明索引使用的越充分。

ref

显示索引的哪一列被使用了。

注意不要同前面的 访问类型 type中的ref造成混淆（表示使用了 非唯一性索引扫描）

rows

rows 列显示 MySQL 认为它执行查询时必须检查的行数。越少越好！

Extra

额外的信息，常见的内容和说明如下：

  • using filesort : 无法利用索引进行排序，只能利用排序算法进行排序，会消耗额外的空间

  • using temporary:建立临时表来保存中间结果，查询完成之后把临时表删除。

  • using index: 当前的查询是索引覆盖（要查询的内容通过索引完全获取，而不用访问数据表）。如果同时出现using where 表名索引被用来执行索引键值的查找，如果没有，表面索引被用来读取数据 。

  • using where:使用where进行条件过滤。

  • using join buffer:使用连接缓存（ 在JOIN的 Blocked Nested Loop 算法中适用（驱动表的数据加入到了 join buffer中))。
