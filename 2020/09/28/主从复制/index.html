主从复制

发表于 2020-09-28 更新于 2020-09-29 分类于 数据库

概念

一直在说主从复制，这到底是个什么东西？到底是为了什么提出？ 其实从字面上就可以知道， 这是个数据的复制过程， 把一个服务器数据库上的数据 复制 到另外一个或者几个服务器上 ，数据源服务器就是“主”， 数据目的服务器就是“从”。

用处

  1. 实时备份数据，发生意外时，让从库随时接管有故障的主库。
  2. 读写分离，主服务器性能较高，可以用作数据写操作，从服务器做查询读操作。 读写分离提高性能（主从服务器各自负责读写，减缓了锁的争用； 从服务器可以使用MyIsam存储引擎，提高查询速度）

模式

  1. 单向主从（单向数据传输， 主服务器到从服务器）
  2. 双向主从 （双向数据传输， 服务器互为对方的master 和 slave）
  3. 级联主从 （从服务器后面接着 从服务器）
  4. 一主多从 （一台主服务器 多态从服务器）
  5. 多主一从

原理

拿单向主从复制举例， 复制的过程主要涉及三个线程：

  1. 主机的 binlog线程（IO Dump Thread)： 把主库接收到的SQL记录存入 bin_log中。
  2. 从机的 IO Thread： 负责请求主机的binlog日志，将得到的内容写入 从机的 Relay log(中继日志)中。
  3. 从机的 SQL Thread：重做Relay log 中的SQL语句。实现内容复制。

可见，传递的不是直接的数据，而是传递的SQL操作，在从机中重做了。

这三个过程中，前两个都是 顺序 IO，速度较快。 而最后一步则是 随机IO，速度较慢。

延迟

由于种种原因，主从服务器上的数据不可能完全没有延迟，延迟的原因主要有以下几点 ：

  1. 从服务器的 SQL Thread只有单线程执行任务。

  2. 大事务执行： binlog的写入必须在事务完成之后，如果事务执行时间过长，那么就会造成在主机写入 binlog时长时间等待，造成延迟。

  3. 主库写binlog是顺序IO， IO Thread请求binlog也是顺序IO， 而SQL Thread执行是随机IO，效率低。

  4. 网络抖动。

  5. 从服务器性能较低。

解决延迟

并行复制

上面提到，最主要的原因就是 在 SQL Thread执行阶段效率低，且只有单线程，所以针对这个缺点提出了 并行复制的方案。

所谓的并行复制，就是在中间添加了coordinator分发过程，将任务分发给多个worker同时执行（人多力量大，很简单的道理）。worker是真正的执行者。

规则

并行操作的时候，如果不遵守一定的规则，就会造成执行混乱， 那么应该遵守什么样的规则呢？

  1. 同一行的多个事务必须要分发到同一个woker中执行（ 若分发到不同的 woker, 可能会造成执行顺序不遵守原来的顺序，产生错误）
  2. 同一个事务不能被拆开到不同的事务中， 必须在同一个事务中。

分组提交二阶段提交

我们知道，磁盘的读写速度相较于CPU 或者主存而言，速度极慢，而且过多的IO 也会造成性能下降，所以写日志的时候，如果不进行相应的改良，那么将比较耗费时间和资源。

分组的提交：为了尽量减少 IO的次数，同时保证数据的 持久性和一致性，提出了分组的提交的概念。 大致思想就是写redolog和 binlog的时候，将操作以组的方式 进行提交。

redolog是 存储引擎级别的，里面记录的是一些具体的物理操作，比如去磁盘的哪个地方执行什么操作

而binlog是属于 服务层次级别的 ， 里面记录的是SQL逻辑操作。

参考图解分组提交写的，

Flush 阶段 (图中第一个渡口)

  • 首先获取队列中的事务组
  • 将Redo log中prepare阶段的数据刷盘(图中Flush Redo log)
  • 将binlog数据写入文件，当然此时只是写入文件系统的缓冲，并不能保证数据库崩溃时binlog不丢失 (图中Write binlog)
  • Flush阶段队列的作用是提供了Redo log的组提交。
  • 如果在这一步完成后数据库崩溃，由于协调者binlog中不保证有该组事务的记录，所以MySQL可能会在重启后回滚该组事务

Sync 阶段 (图中第二个渡口)

  • 这里为了增加一组事务中的事务数量，提高刷盘收益，MySQL使用两个参数控制获取队列事务组的时机：

​ binlog_group_commit_sync_delay=N：在等待N μs后，开始事务刷盘(图中Sync binlog)

​ binlog_group_commit_sync_no_delay_count=N：如果队列中的事务数达到N个，就忽视binlog_group_commit_sync_delay的设置，直接开始刷盘(图中Sync binlog)

  • Sync阶段队列的作用是支持binlog的组提交
  • 如果在这一步完成后数据库崩溃，由于协调者binlog中已经有了事务记录，MySQL会在重启后通过Flush 阶段中Redo log刷盘的数据继续进行事务的提交

Commit 阶段 (图中第三个渡口)

  • 首先获取队列中的事务组
  • 依次将Redo log中已经prepare的事务在引擎层提交(图中InnoDB Commit)
  • Commit阶段不用刷盘，如上所述，Flush阶段中的Redo log刷盘已经足够保证数据库崩溃时的数据安全了
  • Commit阶段队列的作用是承接Sync阶段的事务，完成最后的引擎提交，使得Sync可以尽早的处理下一组事务，最大化组提交的效率

这种实现的优势在于三个阶段可以并发执行，从而提升效率。注意prepare阶段没有变，还是write/sync redo log.

由上面可知，binlog的提交方式是二阶段提交（flush阶段是 binlog 的prepare阶段，完成数据的一些操作， sync阶段是 commit阶段，进行提交，只不过是多了分组操作。

二阶段提交：

可以保证数据的一致性。

  1. prepare阶段：执行数据修改和日志记录等处理，处理完成后只是把事务的状态改成”可以提交”,然后把结果返回给事务管理器。

  2. commit阶段： 如果在第一阶段内有任何一个数据库的操作发生了错误，或者事务管理器收不到某个数据库的回应，则认为事务失败，回撤所有数据库的事务。数据库服务器收不到第二阶段的确认提交请求，也会把”可以提交”的事务回撤。如果第一阶段中所有数据库都提交成功，那么事务管理器向数据库服务器发出”确认提交”请求，数据库服务器把事务的”可以提交”状态改为”提交完成”状态，然后返回应答。

  3. binlog就是二阶段提交。
